# NutriWise — ML: Semantic Embedding Re-Ranker

Reproducible experiment comparing a **fuzzy string baseline** against a **multilingual embedding
re-ranker** for matching receipt lines to BLS 4.0 nutrition entries.

> This is **inference, not training** — we use a pre-trained model and measure it. The product stays
> rule-based / LLM-hybrid ("keep rules where they generalize").

## Data used

- `../bls_off_judgments.json` — 76 items, 227 occurrences, 3-LLM judge panel. Provides the **gold standard**
  (the judge-confirmed correct BLS match per receipt line).
- `../BLS_data/BLS_4_0_Daten_2025_DE.xlsx` — 7140 BLS food names = the candidate corpus.

**Gold set** = the 45 items whose BLS verdict was `correct` or `partially_correct`. For these we know the
correct answer, so we can measure retrieval accuracy.

## Key metric: ExpTop-1

Fuzzy scorers produce many ties (several candidates at an identical top score). Best-case top-1 flatters
them. **ExpTop-1** (expected top-1) breaks ties fairly: with *k* candidates at the top score, a correct one
counts only `1/k`. This is what a system actually delivers when it must auto-pick a single match.

## Result (headline)

| Use case | Best model | Fair top-1 |
|---|---|---|
| **Auto-pick** (system picks 1) | **Hybrid score** (75% WRatio + 25% embedding) | **33%** (vs. 26% fuzzy) |
| **Shortlist** (user picks from top-3/5) | pure Fuzzy WRatio | R@3/R@5 = 56% |
| Standalone embedding search | worst option | 16% |

**Conclusion:** an off-the-shelf embedding is a useful **tie-breaker**, not a primary matcher. It fails on
token traps (`Cola`→`Colabier`) and food-domain nuance, but helps on true synonyms with no character overlap.

## Extended gold set & tiered matching (Step 2)

The 45 curated pairs were extended to the **real receipts** (`../receipts/receipts_queries.json`, 180 line
items). Each unique query was matched against the **full 7,140-name corpus** (not the ranker's own shortlist,
so recall is not inflated), LLM-judged, then human spot-checked.

- **Coverage:** 178 / 180 usable golds — **174 via BLS**, **4 via the OFF fallback** (branded/spice products
  BLS lacks). 2 stay unresolved (a counter fish of unspecified species — an ambiguity we refuse to guess).
- **Label-error rate:** 3% (1/29 spot-checked, BLS + OFF); the one miss (jarred broth type) was corrected.
- **Cleanup effect** (hybrid, extended set): ExpTop-1 **18% → 25%**, R@5 **32% → 47%** from raw → cleaned query.
- **Real receipts are harder than the curated 45** (R@5 47% vs 56%) — brand/quantity/organic noise tokens +
  raw-vs-prepared BLS granularity. On *raw* queries, pure fuzzy even beats the hybrid → embedding is a
  tie-breaker, confirmed on real data.

Labels are tagged with `gold_source` (`bls` / `off`) and never mixed silently — OFF's crowd-sourced nutrient
data is good for *identifying* a product but weaker than BLS for *precise nutrition*.

## OCR recall (upstream of matching)

Matching can only match text OCR actually recovered, so `ocr_eval.py` measures
that ceiling against the real receipt files in `../receipts/`.

```bash
.venv/bin/python3 ml/ocr_eval.py
```

It surfaced 6 landscape phone photos (missing EXIF orientation) that OCR'd to
pure noise — mean recall 61.8%. The fix (confidence-based 90° orientation
recovery, now in `backend/app/services/local_extractor._ocr`) lifted mean recall
to ~93% and added +10 parsed items end-to-end, with zero change on any receipt
that already worked. Remaining `LOW` receipts are a **structured-parser** limit
(ALDI price format), not OCR — the raw text is correct.

## Honest limits

- 45 items — small, wide confidence intervals.
- α tuned on the same set it is evaluated on → optimistically biased (exploration, not a product number).
- Exact-name metric penalizes equivalent BLS alternatives → real usable rate is likely higher.
- No fresh judge verdicts — this is a retrieval benchmark against known gold answers.

## What would actually move the needle (in order)

1. **Tier 0 "learned verified matches"** — collect user confirmations as training pairs. No external database
   needed; the app grows its own labels in production.
2. **Domain-adapted / fine-tuned embedding** on food pairs (needs ~1k–10k labelled pairs — not yet available).
3. **Richer query normalization** (strip quantities/brands/units) before matching.
